# AWSを無料（または低コスト）で学習するためのガイド

AWSを学習目的で触る場合は、**無料枠（Free Tier）を活用しつつ、請求ガードを先に整える**のが最も安全です。本書では「継続的に学習したい」前提で、事故（想定外の課金）を起こしにくい進め方をまとめます。

---

## 1. AWSの「無料」の考え方（まずここが重要）

AWSの無料には大きく2種類あります。

### 1.1 常時無料（Always Free）
アカウント保有中、毎月一定量まで無料になる枠です。  
少量の監視・サーバレス・データベース等で見かけることが多いタイプです。

### 1.2 12か月無料（12 Months Free）
新規アカウント作成から**12か月間のみ**、一定量まで無料になる枠です。  
代表例として、EC2の小さめインスタンス枠などがあります。

> 注意：いずれも「無料＝無制限」ではありません。  
> 無料枠対象外のサービスもあり、上限を超えると課金されます。

---

## 2. 事故防止：最初に入れるべき請求ガード（最優先）

継続利用では「たまにしか触らない月」が発生しがちで、そこで止め忘れ事故が起きやすくなります。以下を最初に設定してください。

### 2.1 AWS Budgets（予算）を作る
- 月額上限を**少額**に設定（例：少額のUSD相当）
- 予算の **50% / 80% / 100%** などでメール通知

### 2.2 請求アラート（Billingアラート）を有効化
「課金が発生しそう／発生した」を早期検知できます。

### 2.3 MFA（多要素認証）を必ず有効化
- ルートユーザーにMFAを設定
- 以後、ルートユーザーは基本使わず、通常作業はIAMユーザー/ロールで行います

### 2.4 リージョンを固定する
最初は**1リージョン**（例：東京）に寄せると、資源の把握が簡単になり、削除漏れも減ります。

---

## 3. 学習用の基本設計：IAMで「学習用ユーザー」を作る

- ルートユーザーで日常作業しない
- IAMで学習用のユーザー/ロールを作成する
- 権限はできるだけ**必要最小限（最小権限）**にする

> 認証情報（アクセスキー等）は、公開リポジトリや画面共有に載せないことが重要です。

---

## 4. 無料枠で学びやすい「定番サービス」セット

学習が進めやすく、比較的コスト事故が少ない流れです。

- **IAM**：権限の基礎（全ての土台）
- **S3**：ストレージの基本（多くのサービスと連携）
- **CloudWatch**：ログ・メトリクス・アラーム（安全運用に直結）
- **Lambda + API Gateway**：サーバレスで動くものを作りやすい
- **DynamoDB**：小規模のNoSQLを扱いやすい

---

## 5. 課金事故が多いポイント（避け方つき）

### 5.1 EC2の止め忘れ
- 使い終わったら停止だけでなく、学習内容によっては**削除**まで実施

### 5.2 EBS（ディスク）やスナップショットが残る
- インスタンスを削除してもボリュームやスナップショットが残ることがあります
- 不要なら削除する

### 5.3 NAT Gateway / VPC関連
- 意外にコストが出やすいので、最初は避けるのが安全

### 5.4 CloudWatch Logsの出しすぎ
- 大量ログは課金につながるため、学習では控えめに運用する

---

## 6. 学習後の片付け（クリーンアップ）を習慣化する

学習の最後に、以下を確認するチェックリストを持つと効果的です。

- EC2 / RDS / Lambda / API Gateway
- EBS / スナップショット
- Load Balancer
- S3バケット（中身が残ると削除できないことがあります）
- CloudWatch Logs（ロググループ・保持期間）

---

## 7. 触りたい領域（インフラ・サーバレス・DB・機械学習）と追加の候補

### 7.1 ご希望の領域
- **インフラ**：VPC / EC2 / Security Group / Systems Manager
- **サーバレス**：Lambda / API Gateway / EventBridge / SQS
- **データベース**：DynamoDB（サーバレス寄り）/ RDS（管理型）
- **機械学習**：ML系サービスは強力ですが、構成によってはコスト管理が難しめ

### 7.2 一緒に学ぶと効率が上がる候補（おすすめ）
- **IAM**：最小権限の考え方は長期で効く
- **CloudWatch**：監視は「安全に触る」ための必須スキル
- **S3**：多くの構成の中心になる
- **IaC（CDK / CloudFormation）**：作った環境をコードで再現でき、片付けも簡単になる  
  継続学習で「何を作ったか分からない」問題を防ぎやすい

---

## 8. 事故りにくい順の学習ロードマップ（目標未定でも進めやすい）

目標が固まっていなくても、最終的に“作品”が残りやすい順序です。

### Phase A：基礎と安全（IAM / S3 / CloudWatch）
**狙い**：安全に触れる土台作り＋ログを追える状態  
**ミニ成果物**：S3にファイルを置き、CloudWatchでログ/メトリクスを確認できる

### Phase B：サーバレス（Lambda + API Gateway + DynamoDB）
**狙い**：低コストで「動くアプリ」を作る定番構成  
**ミニ成果物**：メモAPI（作成/一覧/削除など）

### Phase C：インフラ（VPC + EC2：課金しない作法込み）
**狙い**：ネットワークとサーバの基本理解  
**方針**：
- 「起動したらその日のうちに削除」を基本にする
- 長期稼働が必要なら、常用はサーバレス側に寄せ、EC2はスポット学習にする

### Phase D：データベース拡張（RDS）
**狙い**：RDBの管理型（バックアップ/接続/運用の要点）を理解  
**注意**：構成によっては継続課金になりやすいので、必要になってからでOK

### Phase E：機械学習（まずは低コストの触り方）
**結論**：MLは選定を誤るとコストが出やすい領域です。  
**安全な始め方**：
- まずはローカルで学習し、AWSは「推論をAPI化」など小さく使う
- ML系のジョブや推論エンドポイントは、作ったら**必ず削除**（停止では不十分な場合あり）

---

## 9. 継続学習に向いた運用スタイル（おすすめ）

### 9.1 常用は「サーバレス＋小さなデータ」に寄せる
触る頻度が落ちてもコストが跳ねにくく、再開もしやすいです。

### 9.2 インフラ・MLは「触った日に片付ける」前提
EC2 / RDS / MLエンドポイントは置きっぱなしが危険です。

### 9.3 IaCを早めに導入する
「作る→壊す→また作る」を簡単にし、継続学習の再現性を上げます。

---

## 10. 毎回の終了時チェック（最重要の再掲）

学習終了時、最低限ここを確認すると事故を大きく減らせます。

- EC2：インスタンスが残っていないか
- EBS：ボリューム/スナップショットが残っていないか
- RDS：DBが動いていないか
- NAT Gateway / Load Balancer：作成していないか（作ったなら削除）
- ML系：エンドポイントやノートブック等が残っていないか
- CloudWatch Logs：ログが膨らみすぎていないか

---

## 11. 目標が未定のときに選びやすい「最小の目標」案

まずはどれか1つで十分です（後から差し替え可能）。

1. **自分用メモAPI**（API Gateway + Lambda + DynamoDB）
2. **静的サイト＋簡易バックエンド**（S3 + CDN + Lambda等）
3. **監視テンプレ**（CloudWatchアラーム + 通知 + ログ設計）
